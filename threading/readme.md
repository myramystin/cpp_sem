# Процессы

## Base

процесс выделяется ОС для вашей программы 

у каждого процесса есть prcoessID (pid), при желании его можно убить. У каждого процесса есть родительский процесс 

каждый процесс имеет изолированное адресное пространство (ПАМЯТЬ)

процесс владеет памятью, файлами, сокетами

каждый процесс думает, что у него есть вся память компьютера. Это называется виртуальная адресация.

два разных процесса могут иметь одинаковые виртуальные адреса, но это будет разная физическая память. Они не мешают друг другу.

ОС — это диспетчер. Она переключает процессы на CPU очень быстро (тысячи раз в секунду). Это называется вытесняющая многозадачность.

## Как процессы общаются?

стандартный способ - pipe (канал) (примеров не будет)

Pipe — это однонаправленный канал передачи данных между процессами. Представьте водопроводную трубу: вода входит с одного конца и выходит с другого.

## Зомби

Если дочерний процесс завершился, а родитель не вызвал wait(), дочерний становится зомби (zombie process). Он занимает запись в таблице процессов.

## Проблемы 


Дорогое создание -fork() копирует всё адресное пространство
Дорогая коммуникация -Нужен pipe/socket/... для передачи данных
Дорогой context switch	- Смена процесса — тяжёлая операция для ОС

# Потоки

## Base

У каждого потока есть своё:

- Stack (стек вызовов и локальные переменные)
- Регистры CPU
- Счётчик команд (где сейчас выполнение)

Потоки делят между собой:

- Heap (динамическая память)
- Глобальные переменные
- Код программы
- Файловые дескрипторы

ТЕ память не изолированная! -> надо следить за конкуррентным доступом 

## CPP

Как дождаться завершения процесса? 
```cpp
// Вариант 1: join() — ждём завершения потока
// Главный поток блокируется здесь, пока t1 не завершится

std::thread t1(background_task);
t1.join();

// Вариант 2: detach() — отпускаем поток "в свободное плавание"
// Главный поток НЕ ждёт t2

std::thread t2(background_task);
t2.detach();

// ОПАСНОСТЬ: если main() завершится раньше t2,
// то t2 будет принудительно остановлен!
```


## Синхронизация и датарейсы

Почему возникают: counter++ — это НЕ одна операция! Это три:

```
1. LOAD:   Прочитать counter из памяти в регистр
2. ADD:    Прибавить 1 к регистру  
3. STORE:  Записать регистр обратно в память
```

### Mutex

Mutex (mutual exclusion — взаимное исключение) — это примитив синхронизации, который гарантирует, что только один поток может выполнять критическую секцию в один момент времени.

```
Поток 1:                    Поток 2:
lock()   ← получил замок    lock()   ← ЖДЁТ (замок занят)
counter++                   ...
unlock() ← отдал замок      lock()   ← получил замок!
                            counter++
                            unlock()
```

Реализация через атомарные операции на уровне железа

```
Каждый lock() и unlock() — это системный вызов в ядро ОС.

Пользовательский код          Ядро ОС
      │                           │
    lock() ──────────────────────▶│
      │        (переключение      │  проверить mutex
      │         контекста!)       │  поставить в очередь
      │◀────────────────────────── │
      │                           │
   [работает]                     │
      │                           │
   unlock() ────────────────────▶ │
      │        (переключение      │  освободить mutex
      │         контекста!)       │  разбудить ждущих
      │◀────────────────────────── │
```

### Lock_guard

Проблема с lock()/unlock() вручную: если между ними произойдёт исключение — unlock() никогда не вызовется, и программа зависнет (deadlock).

std::lock_guard — RAII-обёртка, которая автоматически освобождает mutex при выходе из области видимости.

unique_lock — более мощная версия lock_guard, позволяет делать lock/unlock вручную

### Атомарность

Для простых операций (инкремент, сравнение) mutex — избыточен. Есть более эффективный инструмент — std::atomic.

Атомарная операция — операция, которая выполняется целиком или не выполняется вовсе. Её нельзя прервать посередине.

### Deadlock

Одна из самых страшных ошибок при работе с mutex — deadlock (взаимная блокировка).

```
Поток 1 ждёт mutex B (который держит Поток 2)
Поток 2 ждёт mutex A (который держит Поток 1)
→ Оба ждут вечно!

    Поток 1          Поток 2
       │                │
    lock(A) ✓        lock(B) ✓
       │                │
    lock(B) ← ЖДЁТ  lock(A) ← ЖДЁТ
       │                │
       :(               :(
```

### Memory order 

Компилятор жесточайше оптимизирует код 

```
// Ваш код:
int a = 0, b = 0, temp = 0;

a = 1;
temp = b;  // читаем b
a = 2;

// Компилятор видит: первый a=1 НЕМЕДЛЕННО перезаписывается a=2
// Зачем делать две записи? Оптимизируем:

// После оптимизации:
temp = b;  // компилятор переставил!
a = 2;     // только одна запись в a
```

Для однопоточного кода это абсолютно корректно — результат одинаковый. Но для многопоточного — другой поток может увидеть промежуточные состояния.

```
namespace std {
    enum class memory_order {
        relaxed,   // Нет никаких гарантий порядка
        consume,   // Зависимые операции упорядочены (deprecated в C++17)
        acquire,   // Никакое чтение/запись не поднимется выше
        release,   // Никакая запись не опустится ниже
        acq_rel,   // acquire + release одновременно
        seq_cst    // Полная последовательная согласованность
    };
}
```
