# templates

Шаблоны в C++ — это механизм метапрограммирования времени компиляции. Важно понимать, что шаблон сам по себе не является кодом — это инструкция компилятору, как создать код.

Процесс работы с шаблонами:

Парсинг — компилятор читает определение шаблона, но не генерирует код

Инстанцирование — когда вы используете шаблон с конкретным типом, компилятор создаёт специализированную версию

Компиляция — сгенерированный код компилируется как обычный C++ код

```cpp
template<typename T>
T add(T a, T b) {
    return a + b;
}

int main() {
    // Здесь компилятор создаёт функцию add<int>
    int x = add(5, 3);
    
    // Здесь компилятор создаёт функцию add<double>
    double y = add(5.5, 3.2);
    }
```

Шаблоны обычно должны быть определены в заголовочных файлах, потому что компилятор должен "видеть" полное определение в момент инстанцирования.

## SFINAE + type_traits

SFINAE = Substitution Failure Is Not An Error

Основная идея: Когда компилятор пытается подставить типы в шаблон и это приводит к ошибке, компилятор не останавливается с ошибкой, а просто исключает этот вариант из рассмотрения и пробует другие перегрузки.

## Концепты

В C++ шаблоны всегда были "утиной типизацией" (duck typing) во время компиляции: "Если это выглядит как утка и крякает как утка, то это утка". Компилятор пытался инстанцировать шаблон с переданным типом, и если где-то внутри использовалась несуществующая операция — возникала ошибка.

Фундаментальные проблемы этого подхода:

- Отложенная проверка — ошибки обнаруживаются глубоко внутри шаблона, а не на границе интерфейса
- Неявные требования — что именно требуется от типа T, можно узнать только читая всю реализацию
- Cascade ошибок — одна проблема порождает десятки строк ошибок компиляции
- Отсутствие контракта — нет формального способа описать, что требуется от типа

Математически, концепт — это предикат над типами: функция, которая принимает один или несколько типов и возвращает true или false.

Фактически, концепт - это формальная спецификация интерфейса. Он описывает синтаксические и семантические требования к типу без привязки к конкретной реализации.

Requires-выражение — это compile-time вычисление, которое проверяет, можно ли выполнить набор операций. Важно понимать, что код внутри requires никогда не выполняется — компилятор только проверяет, был бы он валидным.

Это похоже на виртуальную машину компилятора, которая симулирует выполнение кода без реального выполнения.

### Уровни проверки

- Уровень 1: Синтаксическая проверка
Компилятор проверяет, существуют ли необходимые операции, методы, типы-члены. Это чисто синтаксическая валидация — "можно ли написать этот код".

- Уровень 2: Проверка типов выражений
Компилятор проверяет не только наличие операций, но и типы их результатов. Например, не просто "есть ли оператор плюс", а "возвращает ли оператор плюс правильный тип".

- Уровень 3: Семантические ограничения (опциональный, не проверяется автоматически)
Документированные, но не проверяемые компилятором требования — например, что операция сравнения должна быть транзитивной.

Requires-выражение — это compile-time вычисление, которое проверяет, можно ли выполнить набор операций. Важно понимать, что код внутри requires никогда не выполняется — компилятор только проверяет, был бы он валидным.

### Механизм проверки концептов
- Этап 1: Subsumption и частичное упорядочивание
Когда компилятор видит несколько перегрузок с разными концептами, он должен выбрать "самую специфичную". Это делается через механизм subsumption (поглощения).

    Концепт A поглощает концепт B, если:

    Все требования B являются подмножеством требований A
    A более строгий, чем B
    Это создаёт частичное упорядочивание концептов по специфичности.

- Этап 2: Ранняя проверка (Early Checking)
Одно из ключевых отличий концептов от SFINAE — двухфазная проверка шаблонов:

    Фаза 1: Проверка определения шаблона
    Компилятор проверяет, что внутри шаблона используются только операции, гарантированные концептом. Это происходит до инстанцирования, на основе только требований концепта.

    Фаза 2: Инстанцирование
    Когда шаблон инстанцируется с конкретным типом, происходит обычная компиляция.

    Это фундаментальное улучшение: ошибки ловятся на границе абстракции, а не внутри реализации.

    Этап 3: Диагностика
    Когда тип не удовлетворяет концепту, компилятор точно знает:

    Какой концепт не удовлетворён
    Какое конкретное требование нарушено
    В каком месте произошло нарушение
    Это возможно потому, что концепт — это структурированная спецификация, а не просто попытка компиляции (как в SFINAE).

// Subsumption


template<typename T>
concept Integral = std::is_integral_v<T>;

template<typename T>
concept SignedIntegral = Integral<T> && std::is_signed_v<T>;

// SignedIntegral более строгий, чем Integral
// При перегрузке выберется более специфичная версия

template<Integral T>
void process(T value) {
    std::cout << "Integral\n";
}

template<SignedIntegral T>  // Более специфичная
void process(T value) {
    std::cout << "Signed Integral\n";
}

process(5);           // Signed Integral (int signed)
process(5u);          // Integral (unsigned)
